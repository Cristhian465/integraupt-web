Problemática

🧩 Fragmentación de sistemas

Cada área (espacios, incidencias, salud, deportes, psicología, etc.) maneja procesos por separado, sin un sistema único que centralice la información.

📝 Procesos manuales

Reservas de ambientes, reportes de incidencias y control de sanciones se gestionan de forma manual o en hojas de cálculo, generando errores y demoras.

🔑 Múltiples accesos

Estudiantes, docentes y administrativos no cuentan con un punto de acceso único; cada servicio requeriría credenciales distintas.

📊 Falta de trazabilidad

No existe un registro centralizado (auditoría) de quién hizo qué cambio y cuándo, dificultando el control administrativo.

Objetivos del proyecto

  • Centralizar la gestión académica y administrativa en una sola plataforma web.
  • Permitir inicio de sesión único (incluyendo SSO con Google) para todos los perfiles de usuario.
  • Digitalizar procesos clave: reservas de espacios, incidencias, sanciones, salud y bienestar estudiantil.
  • Brindar reportes y auditoría en tiempo real para la toma de decisiones.
  • Construir una arquitectura escalable, capaz de crecer en módulos sin afectar al resto del sistema.

Solución propuesta

✅ Integra UPT: plataforma web modular

Un sistema web único (SPA) que consume múltiples microservicios backend especializados, cada uno encargado de un dominio del negocio universitario, todo accesible desde un solo dominio y un solo login.

Un solo acceso

Login tradicional + Google SSO para estudiantes, docentes y administrativos.

Módulos independientes

Cada funcionalidad vive en su propio microservicio, desplegable y escalable por separado.

Datos centralizados

Base de datos consolidada con catálogos compartidos (usuarios, escuelas, roles).

Flujo de autenticación (Google SSO)

  1. El usuario presiona "Iniciar sesión con Google" en el frontend.
  2. El frontend redirige al microservicio servicio-login (Laravel Socialite).
  3. El backend redirige a Google OAuth para autenticación.
  4. Google devuelve el control al backend mediante un callback autorizado.
  5. El backend valida que el correo institucional exista y esté activo, genera un token de sesión y redirige al frontend.
  6. El frontend valida el token y construye la sesión del usuario.

Las URLs de cada entorno (local / producción) se resuelven dinámicamente mediante variables de entorno, evitando URLs fijas en el código.

Seguridad y control de acceso

Autenticación dual

Login tradicional (usuario/contraseña) y Google SSO, según el tipo de cuenta del usuario.

Roles y permisos

Perfiles diferenciados: estudiante, docente, administrativo — cada uno con accesos y filtros propios (por escuela, rol, etc.).

Auditoría

Registro de acciones administrativas (creación, edición, activación/inactivación de usuarios y recursos).

HTTPS de extremo a extremo

Certificados SSL (Let's Encrypt) en el dominio público; todo el tráfico cifrado en producción.

Despliegue e infraestructura

  • Cada microservicio y el frontend corren en su propio contenedor Docker, orquestados con docker-compose.yml.
  • Un Apache en el VPS actúa como reverse proxy: expone un solo dominio público y enruta internamente /svc/<servicio>/ a cada contenedor.
  • Variables de entorno separan configuración de desarrollo local (localhost) y producción (dominio real), sin mezclar código.
  • Certificado SSL/TLS automático y renovable mediante Let's Encrypt / Certbot.
  • Base de datos poblada con información real de la universidad (usuarios, escuelas, docentes, estudiantes).
